# SPDD-анализ: Разделение gRPC Server и Client на отдельные пакеты

## Исходное бизнес-требование

мне нужно разделить server и клиент для grpc

---

## Идентификация доменных концепций

### Существующие концепции (из кодовой базы)

- **`grpc.Server`** (`grpc/server.go`): управляемый gRPC-сервер, реализует `service.Service` + `service.Shutdown`. Создаёт `*grpc.Server` внутри `NewServer(opts ...Option)`, экспонирует его через `GRPCServer()` для регистрации сервисов до `Start`. Опции: `WithHost`, `WithPort`, `WithLogger`, `WithServerOptions`.

- **`grpc.Client`** (`grpc/client.go`): управляемый gRPC-клиент, реализует `service.Service` + `service.Shutdown`. Соединение устанавливается в `Start`, закрывается только в `Stop`. Опции: `WithDialOptions`, `WithClientLogger`.

- **Оба типа живут в одном пакете `grpc`** — это и есть корень требования. Из-за общего пакета уже сейчас понадобилось развести имена: `Option` (сервер) vs `ClientOption` (клиент), `WithLogger` (сервер) vs `WithClientLogger` (клиент) — иначе имена конфликтовали бы в рамках одного пакета.

- **`service.Service` / `service.Shutdown`** (`service/service.go`): контракты жизненного цикла, которым подчиняются оба типа. Разделение пакетов их не затрагивает — оба контракта пакето-независимы.

- **Прецедент в кодовой базе — `httpserver` / `httpclient`**: HTTP-аналоги уже реализованы как **два отдельных top-level пакета**, а не один `http`-пакет с `HttpServer`/`HttpClient` внутри. Комментарии к `GrpcServer`/`GrpcClient` в `spdd/analysis/...-toolkit-roadmap.md` прямо называют их «зеркалом HttpServer»/«зеркалом HttpClient» — то есть по замыслу они должны были повторять именно эту структуру, но при реализации (см. `spdd/prompt/...-toolkit-aiomisc-primitives.md`, раздел Approach: «`GrpcServer` и `GrpcClient` живут в одном пакете `grpc`») было сознательно выбрано объединение в один пакет. Это единственное расхождение с иначе последовательной конвенцией «один пакет — один компонент» (`entrypoint`, `httpserver`, `httpclient`, `periodic`, `workerpool`, `healthserver`).

### Новые концепции, которые нужно добавить

- **Два отдельных пакета вместо одного `grpc`**: разнести `Server` и `Client` физически, по аналогии с `httpserver`/`httpclient`. Требует решения по именованию (см. Strategic Approach) и обновления путей импорта.

- **Схлопывание раздельных имён опций обратно к общим**: после разделения `ClientOption` → `Option`, `WithClientLogger` → `WithLogger` в клиентском пакете больше не нужны — коллизии не будет, т.к. `Server` и `Client` окажутся в разных пакетах. Это восстанавливает симметрию с `httpserver.Option`/`httpclient.Option`, у каждого из которых сейчас единообразный `WithLogger`.

### Ключевые бизнес-правила

- **Независимая подключаемость компонентов** (Requirements прошлого prompt-файла: «Каждый примитив должен быть независимо подключаемым и работать как в составе Entrypoint, так и автономно») — управляет тем, что потребитель, которому нужен только `Client`, не должен тянуть код `Server` в тот же compile-unit и наоборот. Сейчас оба живут в одном пакете `grpc`, что нарушает дух этого правила буквально (хотя не физического веса зависимостей — `google.golang.org/grpc` всё равно общий).
- **Явная конфигурация без магии** — оба типа продолжают конфигурироваться через functional options; правило не меняется разделением пакетов.
- **Context — сигнал остановки** — оба типа управляются через `service.Service`/`service.Shutdown`; контракт не меняется разделением.
- **Библиотека уже опубликована и используется другими сервисами** (`gokit-services` — переиспользуемый тулкит для нескольких Go-микросервисных проектов, коммит с текущей структурой `grpc/` уже запушен в `origin/main`, тегов версий пока нет) — любое изменение пути импорта `github.com/DjaPy/gokit-services/grpc` — это breaking change для потребителей, которые уже сделали `go get` на этот коммит/ветку.

## Стратегический подход

### Направление решения

Вынести `Server` и `Client` из общего пакета `grpc` в два отдельных top-level пакета — по прямой аналогии с уже существующей парой `httpserver`/`httpclient`. Это устраняет искусственное разведение имён (`Option`/`ClientOption`, `WithLogger`/`WithClientLogger`), появившееся исключительно из-за совместного размещения, и приводит gRPC-примитивы к тому же уровню единообразия, что и HTTP-примитивы. Поток данных не меняется — оба типа как реализовывали `service.Service`/`service.Shutdown` и подключались в `Entrypoint` независимо, так и продолжат; меняется только границы пакета и путь импорта.

### Ключевые решения дизайна

- **Именование новых пакетов**: `grpcserver` + `grpcclient` (слитно, как единое слово) vs `grpc/server` + `grpc/client` (вложенные под общий `grpc/`) vs `grpcserver`/`grpcclient` с сохранением пустого `grpc/` как namespace-заглушки.
  → **Рекомендация**: `grpcserver` + `grpcclient`, плоские top-level директории. Совпадает буква в букву с существующей конвенцией именования пакетов в `CLAUDE.md` («строчные, одним словом»; `grpc` там был явно помечен как единственное исключение «название протокола» — после разделения это исключение исчезает, и правило становится по-настоящему единообразным без специальных случаев). Вложенный вариант `grpc/server`+`grpc/client` добавляет уровень директорий, не используемый нигде больше в проекте (`httpserver` не лежит в `http/server`).

- **Судьба разведённых имён опций**: оставить `ClientOption`/`WithClientLogger` в новом `grpcclient` (минимальный диф) vs вернуть к `Option`/`WithLogger` (полная симметрия с `httpserver`/`httpclient`, оба из которых используют `Option`/`WithLogger` без вариаций).
  → **Рекомендация**: вернуть к `Option`/`WithLogger` в обоих новых пакетах — коллизия имён была единственной причиной их разведения; убрав причину, стоит убрать и следствие, иначе останется необъяснимая асимметрия с HTTP-парой.

- **Судьба текущего пакета `grpc`**: удалить полностью (breaking) vs оставить как deprecated re-export shim (`type Server = grpcserver.Server`, и т.п.) на переходный период.
  → **Рекомендация**: удалить полностью, без shim. Библиотека без тегов версий, без формального semver-контракта; проект явно избегает compatibility-обёрток («Avoid backwards-compatibility hacks» — установленный для этого репозитория принцип). Если у потребителей уже есть код на `.../grpc`, это разовая правка двух строк импорта + пары идентификаторов (`grpc.Server` → `grpcserver.Server` и т.п.) — дешевле, чем поддерживать shim.

- **Тесты**: разнести `grpc/server_test.go` → `grpcserver/server_test.go`, `grpc/client_test.go` → `grpcclient/client_test.go`. Текущие тесты клиента уже зависят от сервера (`grpc/client_test.go` импортирует и поднимает `gkgrpc.NewServer` для проверки клиента) — при разделении `grpcclient_test` (`package grpcclient_test`) получит **прямую зависимость на `grpcserver`** в тестовом коде. Это осознанный трейд-офф: тестовая связанность (typical и приемлемо — так же зависят друг от друга тесты `httpclient` от `httptest`), а не связанность production-кода.

### Рассмотренные альтернативы

- **Оставить всё в одном пакете `grpc`, просто переименовать опции обратно и утверждать, что коллизии больше нет**: невозможно — `Option`/`WithLogger` не могут одновременно существовать для `Server` и `Client` в одном пакете без коллизии; текущая раздвоенность (`ClientOption`) — это и есть цена совместного размещения, обойти её без физического разделения пакетов нельзя. Собственно это и есть причина, по которой пользователь просит разделение.
- **Разделить только опции (создать под-пакет `grpc/option`), но оставить `Server`/`Client` в одном пакете `grpc`**: не решает заявленную проблему — пользователь просит разделить именно server и client, а не только конфигурацию.
- **Submodule `gokit-services/grpc` (отдельный go.mod)**: рассматривался ранее в roadmap-анализе как способ не утяжелять модуль gRPC-зависимостью для потребителей, которым он не нужен — другая ось разделения (модуль, не пакет), решает другую проблему (вес зависимостей), не связана с текущим требованием и не решает его. Не относится к этому анализу.

## Анализ рисков и пробелов

### Неоднозначности требования

- **Формулировка предельно краткая, явных Acceptance Criteria нет.** «Разделить server и клиент» можно прочитать как (а) разнести по разным пакетам (принято как основная гипотеза выше, согласуется с прецедентом `httpserver`/`httpclient`), (б) разнести по разным файлам в том же пакете `grpc` (`server.go`/`client.go` — но это уже так сделано, требование было бы бессмысленным), (в) разнести на уровне модулей (`go.work`/submodule) — маловероятно, эта опция уже рассматривалась и отклонена в roadmap-анализе по другой причине. **Нужно подтверждение** у пользователя, что имеется в виду именно (а) — физическое разделение пакетов.
- **Название новых пакетов не указано пользователем** — предложено `grpcserver`/`grpcclient` по аналогии с `httpserver`/`httpclient`, но это решение, а не факт из требования; нужно подтверждение.
- **Судьба старого пути импорта `.../grpc` не указана** — предложено удаление без deprecation-shim (см. Key design decisions), но это тоже decision, требующий подтверждения, учитывая что коммит уже в `origin/main`.

### Граничные случаи

- **Общий тестовый файл, проверяющий совместную работу Server+Client** (например, если в будущем появится интеграционный тест «клиент подключается к серверу и делает RPC» на уровне выше обоих пакетов) — после разделения такой тест физически не может жить ни в `grpcserver_test`, ни в `grpcclient_test` без того, чтобы один из пакетов не тянул другой в тестовую зависимость. Сейчас `TestGrpcClient_ConnAvailableAfterStart` и `TestGrpcClient_StopClosesConn` уже поднимают `grpc.NewServer` внутри `grpc/client_test.go` — после разделения это станет тестовой зависимостью `grpcclient_test` → `grpcserver`, что нормально, но нужно явно принять.
- **`CLAUDE.md` описывает `grpc/` как единый пакет** («`grpc/` — gRPC клиент (в разработке)») — документация уже отстаёт от текущего кода (там сервер и клиент, не только клиент) и потребует обновления при любом исходе этого анализа, независимо от решения о разделении.

### Технические риски

- **Breaking change для существующих потребителей**: коммит `f14cd44` с текущей структурой `grpc/` (`Server` + `Client` в одном пакете) уже на `origin/main`. Тегов/релизов нет, но потребители могли зафиксировать зависимость на конкретный коммит или ветку `main` через `go get github.com/DjaPy/gokit-services@<commit>`. Переименование пакета — это изменение публичного API (import path + идентификаторы), которое сломает любую такую сборку при следующем `go get -u`. Митигация: если есть известные потребители — синхронно уведомить; либо выпустить это как отдельный минорный коммит с явным сообщением "BREAKING: split grpc package" в истории.
- **Риск рассинхронизации с уже обновлённым `spdd/prompt/...-toolkit-aiomisc-primitives.md`**: этот prompt-файл был только что синхронизирован (`/spdd-sync`) с текущей структурой (`grpc.Server`/`grpc.Client` в одном пакете, `ClientOption`/`WithClientLogger`). Разделение пакетов потребует повторной синхронизации prompt-файла после реализации — иначе он снова разойдётся с кодом сразу после смены.
- **Нет технических рисков в самой механике переноса** — перенос кода `server.go`→`grpcserver/server.go`, `client.go`→`grpcclient/client.go` синтаксически прост (два независимых файла и так почти не имеют внутренних перекрёстных ссылок друг на друга в production-коде, только в тестах клиента); риск сосредоточен полностью в области совместимости API, не в реализации.
