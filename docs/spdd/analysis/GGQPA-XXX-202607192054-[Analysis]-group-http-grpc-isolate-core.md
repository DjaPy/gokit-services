# SPDD Analysis: Группировка http/grpc по образцу kafka и изоляция core-пакетов (entrypoint, service)

## Original Business Requirement

> необходимо сгруппировать похожим образом grpc и http, как это сделано с kafka, так же отделить логику пакетов entrypoint, service от остальных реализаций

## Domain Concept Identification

Это рефакторинг компоновки самого тулкита — «доменом» здесь выступает структура пакетов библиотеки, а не бизнес-сущности. Концепты — это пакеты и слои.

### Existing Concepts (from codebase)

- **Эталон группировки `kafka/`**: родительский пакет `kafka` (`kafkanet.go`, `consts.go`) держит *разделяемые транспортные примитивы* — `NewDialer`, `TLSSASLConfig`, `SASLMechanism`, probe. Подпакеты `kafka/producer` и `kafka/consumer` импортируют родителя (`kafkanet "…/kafka"`), но **не зависят друг от друга** — producer-only сервис не тянет consumer и его worker pool. Это и есть смысловое ядро паттерна, а не просто вложенность директорий.
- **`httpserver` / `httpclient`**: два независимых пакета верхнего уровня. Проверено: общего кода и перекрёстных импортов между ними нет.
- **`grpcserver` / `grpcclient`**: аналогично — два независимых пакета, без общего кода и перекрёстных импортов (были осознанно разделены, см. `docs/spdd/analysis/…-split-grpc-server-client.md`).
- **`service`**: пакет-контракт. Объявляет интерфейсы `Service`, `Shutdown`, `Prober`. Импортируется `entrypoint`, `healthserver` и примером. Не импортирует ничего из тулкита — это корень графа зависимостей.
- **`entrypoint`**: оркестратор жизненного цикла. Из внутренних пакетов импортирует **только** `service`. Реализации транспортов не импортирует (работает через интерфейс `service.Service`).
- **Прочие реализации**: `healthserver` (строится поверх `httpserver` + `service`), `periodic`, `workerpool` (используется `kafka/consumer`), `dbservice`, `redisservice`.

### New Concepts Required

- **Группа `http/`**: зонтичная директория с `http/server` и `http/client` (перенос `httpserver`/`httpclient`). Открытый вопрос — существует ли смысловой родительский пакет `http` (как у kafka) или это только неймспейс-директория (см. решение D1).
- **Группа `grpc/`**: зонтичная директория с `grpc/server` и `grpc/client`.
- **Core/контрактный слой**: выделенная группа для `entrypoint` + `service` — пакеты «фреймворка» (жизненный цикл + контракты), отделённые от конкретных реализаций транспортов и инфраструктуры.

### Key Business Rules

- **Независимость server/client** (инвариант kafka producer/consumer): после переноса `http/server` и `http/client` (равно `grpc/*`) не должны импортировать друг друга — сервис, которому нужен только клиент, не тянет сервер с Prometheus-метриками, и наоборот.
- **Направление зависимостей core → impl запрещено**: core-слой (`service`, `entrypoint`) не должен зависеть от реализаций. Сейчас это соблюдается (`entrypoint → service` и всё; транспорты — через интерфейс). Рефакторинг обязан это свойство сохранить.
- **Отсутствие поведенческих изменений**: чистое перемещение пакетов и правка import-путей. Публичные API, метрики, сигнатуры не меняются.
- **Отсутствие import-циклов**: `healthserver` зависит от `httpserver` — после группировки эта дуга станет `healthserver → http/server`; нельзя допустить, чтобы группировка создала цикл (например, если бы `http`-родитель что-то тянул обратно).

## Strategic Approach

### Solution Direction

Механический реорг раскладки пакетов в два независимых движения:

1. **Группировка транспортов по образцу `kafka/`**: `httpserver → http/server`, `httpclient → http/client`, `grpcserver → grpc/server`, `grpcclient → grpc/client`. Общий поток: перенос файлов (`git mv` для сохранения истории) → смена `package`-объявлений → массовая правка import-путей у всех потребителей (внутри — `healthserver` и весь `example/`; снаружи — внешние консьюмеры).
2. **Изоляция core**: собрать `entrypoint` + `service` в отдельный слой, визуально и структурно отделённый от реализаций (транспорты + инфра-сервисы).

Поскольку модуль — чистая библиотека pre-1.0, это допустимое breaking-изменение в MINOR-релизе с пометкой **BREAKING** в `CHANGELOG.md` (по политике из `CLAUDE.md`). Суффикс `/v2` в module path **не требуется** (остаёмся в `v0.x`). Следующий тег — `v0.4.0`.

### Resolved Decisions (locked by user)

Открытые вопросы закрыты пользователем — REASONS Canvas строится на этих выборах:

- **D1 → чистый неймспейс.** `http/` и `grpc/` — только директории-зонтики, **без** `.go`-файлов в корне. Родительские пакеты `http`/`grpc` не заводим (искусственный разделяемый пакет там, где нечего разделять, не создаём). Копируем вложенность и инвариант независимости kafka, но не факт наличия файла-родителя.
- **D2 → `server` / `client`.** Пакеты `server` и `client` под `http/` и `grpc/` (`http/server`, `http/client`, `grpc/server`, `grpc/client`). На местах совместного использования — import-алиасы (`httpsrv`, `grpcsrv`).
- **D3 → `core/`.** `core/entrypoint` + `core/service`. Реализации остаются на верхнем уровне (инверсию «всё под зонтик» не делаем).
- **Скоуп → инфра-сервисы не трогаем.** `healthserver`, `periodic`, `workerpool`, `dbservice`, `redisservice` остаются пакетами верхнего уровня. Рефактор ограничен http/grpc-группировкой + выносом core. (D4 закрыт этим же: `healthserver` остаётся наверху, начинает импортировать `http/server`.)
- **D5 → алиасы (без изменений).** Коллизия имени с `net/http` митигируется алиасами импортов; при чистом неймспейсе она к тому же почти не возникает (импортируются `server`/`client`, а не `http`).

Итоговая целевая раскладка:

```
core/
  entrypoint/   (package entrypoint)
  service/      (package service)
http/
  server/       (package server, бывш. httpserver)
  client/       (package client, бывш. httpclient)
grpc/
  server/       (package server, бывш. grpcserver)
  client/       (package client, бывш. grpcclient)
kafka/          (без изменений: kafka + producer/consumer)
healthserver/ periodic/ workerpool/ dbservice/ redisservice/   (без изменений)
```

### Key Design Decisions

- **D1 — Смысловой родительский пакет vs чистый неймспейс (ключевое решение).** У `kafka` родитель несёт разделяемый код; у http/grpc server и client **сегодня не разделяют ничего**. Варианты: (а) только директорная группировка — `http/` без `.go`-файлов, пакеты `http/server` и `http/client` самодостаточны; (б) завести родительский пакет `http`/`grpc` с общими примитивами (для grpc потенциально — общий TLS/dial/interceptor-хелпер; для http — реального общего кода почти нет). → **Рекомендация**: не изобретать искусственный разделяемый пакет там, где его нет. Для http — чистый неймспейс (а). Для grpc — (а) по умолчанию, но зафиксировать как открытый вопрос, есть ли 1–2 общих примитива, которые естественно поднять в `grpc` (иначе пустой родитель — карго-культ формы kafka без её сути). Честнее реализовать *инвариант* kafka (независимость + вложенность), чем копировать наличие файла-родителя.
- **D2 — Имена подпакетов.** kafka использует однословные `producer`/`consumer` (неколлизионные). Прямой аналог — `http/server`, `http/client` (пакеты `server`/`client`). Но `server`/`client` — коллизионно-генеричные на местах вызова. Альтернатива — сохранить имена пакетов `httpserver`/`grpcserver` внутри сгруппированной директории (`http/httpserver`), что уродливо дублирует префикс. → **Рекомендация**: `http/server`, `http/client`, `grpc/server`, `grpc/client` (пакеты `server`/`client`); на местах, где в одном файле встречаются и http-server, и grpc-server, использовать import-алиасы (`httpsrv`, `grpcsrv`).
- **D3 — Куда уходит core.** Варианты: (а) новая группа `core/` c `core/entrypoint` + `core/service`; (б) оставить `service`/`entrypoint` в корне, а *все реализации* убрать под зонтик (`transport/http`, `transport/grpc`, `transport/kafka`, инфра — под свою группу); (в) промежуточный: `lifecycle/entrypoint` + `service` в корне. → **Рекомендация**: (а) `core/` (или `kit/`) для `entrypoint`+`service` — минимальный, читаемый сигнал «это контракт и оркестрация, а не реализация», меняет меньше import-путей, чем перенос всех реализаций сразу. Финальное имя группы — вопрос к пользователю (`core` / `kit` / `app`).
- **D4 — Место `healthserver`.** Это http-based реализация (импортирует `httpserver`). Варианты: оставить как самостоятельный инфра-сервис верхнего уровня; либо втянуть под `http/` как http-производную. → **Рекомендация**: оставить `healthserver` на верхнем уровне среди реализаций (это сервис, а не транспортный примитив); он просто станет импортировать `http/server`. Не расширять скоуп рефактора на переезд инфра-сервисов. +
- **D5 — Коллизия имени пакета `http` со stdlib `net/http`.** Родитель/группа `http` конфликтует по идентификатору с `net/http` на местах совместного использования. → **Рекомендация**: та же митигация, что уже применена для kafka (алиас `kafkanet`) — алиасить импорты (импортировать сразу подпакеты `http/server`, `http/client`, которые не коллизируют по имени `server`/`client`). 2-о

### Alternatives Considered

- **Ничего не группировать, только переименовать для единообразия** — отклонено: не отвечает требованию «сгруппировать как kafka» и не даёт визуального отделения core.
- **Буквально скопировать форму kafka с пустым/искусственным родителем `http`/`grpc`** — отклонено как форма без содержания (см. D1): без разделяемого кода родительский пакет — карго-культ; ценность паттерна kafka в *инварианте независимости*, а не в наличии файла-родителя.
- **Инвертировать: убрать все реализации под `transport/`+`services/`, оставив корень только под core** — жизнеспособно и «чище» концептуально, но радикально расширяет blast radius (правятся импорты *всех* пакетов сразу). Отклонено на этом шаге в пользу меньшего диффа (D3-а); может быть отдельной последующей итерацией.

## Risk & Gap Analysis

### Requirement Ambiguities

Все исходные неоднозначности закрыты пользователем — см. **Resolved Decisions (locked)** выше. Для трассируемости:

- **Скоуп «как с kafka»** → чистый неймспейс, без родительского пакета (D1).
- **«Отделить entrypoint, service»** → вынос в `core/`, реализации остаются в корне (D3).
- **Именование** → группа `core/`; подпакеты `server`/`client` (D2).
- **Судьба инфра-сервисов** (`healthserver`, `periodic`, `workerpool`, `dbservice`, `redisservice`) → остаются как есть, вне скоупа рефактора.

### Edge Cases

- **Коллизия с `net/http`**: пакет/группа `http` конфликтует по идентификатору со stdlib на местах совместного использования — обязателен алиас (D5).
- **Дуга `healthserver → httpserver`**: единственная внутренняя (не-example) зависимость от переносимых транспортов, помимо `kafka/consumer → workerpool`. Её нельзя пропустить при правке импортов, иначе сломается сборка `healthserver`.
- **Синхронное обновление `example/`**: пример — крупнейший потребитель (импортирует `httpserver`, `httpclient`, `grpcserver`, `grpcclient`, `entrypoint`, `service` и все инфра-пакеты). Он обязан быть обновлён в том же коммите, иначе `go build ./...` красный.
- **Пропущенная ссылка при массовой замене**: чисто механический риск — незамеченный старый import-путь. Митигируется `go build ./...` + `go vet ./...` + `golangci-lint run ./...` как гейтом.

### Technical Risks

- **Breaking import-пути для внешних консьюмеров** (`go get …@vX`): все, кто импортирует `httpserver`/`httpclient`/`grpcserver`/`grpcclient`/`entrypoint`/`service`, должны поправить импорты. Митигация: pre-1.0 допускает это в MINOR; обязательная запись в `CHANGELOG.md` под `### Changed` с **BREAKING**; релиз `v0.4.0`; `/v2`-суффикс не нужен.
- **Потеря git-истории файлов** при переносе. Митигация: `git mv` вместо delete+create.
- **Нарушение инварианта независимости** при попытке «удобно» вынести общий код в родитель. Митигация: подтвердить после реорга, что `http/server` не импортирует `http/client` (и grpc-аналог), а core-слой не импортирует реализации — быстрым `go list -f '{{ join .Imports "\n" }}'`.
- **Ложное ощущение готовности**: тесты у пакетов есть (`server_test.go`, `client_test.go`, `entrypoint_test.go` и т.д.) — прогон `go test -race ./...` обязателен как финальная проверка, что перенос ничего не сломал.
