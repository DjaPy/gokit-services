# SPDD-анализ: Go-тулкит для микросервисов — roadmap в стиле aiomisc

## Исходное бизнес-требование

Я делаю библиотеку для go на подобие python aiomisc. Уже заложил фундамент.

---

## Идентификация доменных концепций

### Существующие концепции (из кодовой базы)

- **Service** (`service/service.go`): Универсальный контракт жизненного цикла — `Start(ctx) error` блокирует до полной остановки сервиса; отмена ctx — основной сигнал на завершение. Все управляемые компоненты тулкита реализуют этот интерфейс.

- **Shutdown** (`service/service.go`): Опциональный компаньон Service — `Stop(ctx, cause) error` позволяет сервису выполнить упорядоченную очистку при остановке. Entrypoint проверяет интерфейс через type assertion в рантайме; сервисы без graceful stop реализуют только Service.

- **Entrypoint** (`entrypoint/entrypoint.go`): Единственный оркестратор. Управляет N сервисами параллельно, перехватывает SIGINT/SIGTERM, поддерживает явный вызов `Shutdown()`, соблюдает таймаут при graceful stop. Предоставляет lifecycle-хуки в четырёх фазах.

- **Hook** (`entrypoint/entrypoint.go`): Функция `func(context.Context) error`, регистрируемая в Entrypoint при создании. Четыре фазы: PreStart, PostStart, PreStop, PostStop. Внутри каждой фазы хуки выполняются последовательно.

- **HttpServer** (`httpserver/server.go`): Продакшн-готовый HTTP-сервер, реализующий Service + Shutdown. Оборачивает любой `http.Handler` метриками Prometheus (длительность запроса, inflight-счётчик, размер ответа, счётчик паник) и panic recovery, возвращающим RFC 7807 Problem JSON. SSE и WebSocket работают через проброс Flusher/Hijacker в обёртке responseWriter.

- **HttpClient** (`httpclient/client.go`): Исходящий HTTP-клиент с фиксированным base URL. Дженерик `Do[T]` декодирует JSON-ответ; не-2xx статусы возвращаются как ошибки. Транспорт оборачивается цепочкой middleware (RoundTripperMiddleware) при создании клиента.

### Новые концепции, которые нужно добавить

- **PeriodicService**: Сервис, запускающий пользовательский callback с фиксированным интервалом. Аналог `PeriodService` в aiomisc. Должен корректно завершать выполнение при отмене ctx и определять политику в случае, когда callback занимает больше времени, чем интервал (пропустить тик vs. запустить параллельно).

- **WorkerPool**: Сервис-обёртка над ограниченным пулом горутин для выполнения фоновых задач. Аналог обёртки `ThreadPoolExecutor` из aiomisc, адаптированной для горутин. Требует политики backpressure при заполненном пуле (блокировать / отклонить / вернуть ошибку).

- **GrpcServer**: gRPC-сервер, реализующий Service + Shutdown, — зеркало HttpServer. Пакет `grpc/` существует в виде стаба (`package grpc`, без реализации). Завершение должно различать `GracefulStop` (дренаж in-flight RPC) и `Stop` (немедленно) в рамках таймаута.

- **GrpcClient**: Исходящий gRPC-клиент с управлением lifecycle соединения, — зеркало HttpClient. Стаб `grpc/client.go` — заглушка для этого компонента.

- **HealthServer**: Тонкий сервис, предоставляющий `/healthz` (liveness) и `/readyz` (readiness) эндпоинты. Может быть конструктор-помощником поверх HttpServer или отдельным пакетом. Readiness-эндпоинт потребует нового опционального интерфейса (`Prober`) для опроса готовности других сервисов.

- **CronService**: Сервис, запускающий задачи по cron-расписанию (например, `"0 * * * *"`). Аналог `CronService` из aiomisc. Заблокирован до принятия решения о зависимости на парсер cron-выражений (в stdlib Go нет парсера cron).

### Ключевые бизнес-правила

- **Единый контракт Service**: Каждый управляемый компонент реализует `service.Service`. Реализации без graceful stop допустимы — Shutdown опционален.
- **Entrypoint — единственный оркестратор**: Сервисы не должны управлять жизненным циклом других сервисов напрямую. Координация между сервисами — через разделяемое состояние, каналы или хуки.
- **Никакого глобального состояния по умолчанию**: Prometheus Registerer — единственная точка, близкая к глобальной. В тестах обязательно передавать изолированный `prometheus.NewRegistry()`. Этот паттерн нужно воспроизвести во всех новых сервисах с метриками.
- **Явная конфигурация без магии**: Всё конфигурируется через functional options при создании. Никаких reflection-based injection, init()-сайд-эффектов или глобальных реестров.
- **Context — сигнал остановки**: Сервисы определяют shutdown по отмене ctx в Start. Метод Stop — это слой упорядоченной очистки поверх этого сигнала, а не его замена.

## Стратегический подход

### Направление решения

Тулкит следует философии **компонуемых примитивов** — он предоставляет небольшие ортогональные строительные блоки, которые пользователь соединяет самостоятельно, а не полноценный runtime-фреймворк. Это уже чётко прослеживается в существующем коде и должно сохраняться по мере роста библиотеки. Каждый новый компонент должен быть независимо используемым — без принудительной связанности между, например, PeriodicService и WorkerPool.

Естественный порядок реализации с учётом сложности зависимостей и ценности:

1. **PeriodicService** — максимальная ценность, новых зависимостей нет, только stdlib (`time.Ticker` + горутина)
2. **GrpcServer** — завершает серверную часть; добавляет `google.golang.org/grpc` как зависимость
3. **WorkerPool** — только stdlib, канальная реализация ограниченной очереди
4. **HealthServer** — тонкая обёртка над HttpServer; нужно решить семантику readiness
5. **CronService** — откладывается до принятия решения о зависимости на парсер cron
6. **GrpcClient** — после GrpcServer; заполняет стаб в `grpc/`

### Ключевые архитектурные решения

- **Политика переполнения PeriodicService**: Если callback выполняется дольше интервала, следующий тик можно пропустить (non-overlapping — наиболее распространённая потребность) или запустить параллельно (overlapping — редко, рискованно). → **Рекомендация**: non-overlapping по умолчанию (пропустить при занятости); overlapping — через опцию. Это повторяет поведение aiomisc и безопаснее для большинства нагрузок.

- **Backpressure WorkerPool**: Когда все горутины заняты, Submit может блокировать (естественный backpressure), сразу возвращать ошибку или отбрасывать задачу. → **Рекомендация**: блокировать по умолчанию с context-aware сигнатурой (`Submit(ctx, task) error` возвращает ошибку, если ctx завершён или пул завершается). Это наиболее Go-идиоматичный подход и предотвращает молчаливую потерю задач.

- **gRPC как зависимость модуля**: Добавление `google.golang.org/grpc` увеличивает вес модуля для всех потребителей, включая тех, кто использует только entrypoint + httpclient. → **Рекомендация**: пока держать gRPC в одном модуле (проще, единый `go get`). Если потребители дадут обратную связь — вынести в сабмодуль `gokit-services/grpc` позже. Не усложнять заранее.

- **Readiness-осведомлённость HealthServer**: Готовность (все ли сервисы стартовали и работают?) требует, чтобы сервисы сообщали о своём состоянии. → **Рекомендация**: ввести опциональный интерфейс `Prober` (`Probe(ctx) error`), который HealthServer опрашивает. Сервисы реализуют его при необходимости; HealthServer агрегирует результаты. Это избегает связи HealthServer с внутренностями Entrypoint.

- **Зависимость CronService**: Парсера cron в stdlib нет. Варианты: `robfig/cron` (де-факто стандарт) или минимальная собственная реализация. → **Рекомендация**: отложить CronService до явного решения по политике зависимостей проекта. PeriodicService покрывает большинство повторяющихся задач без парсера. Если добавлять — использовать `robfig/cron/v3` (хорошо поддерживается, минимальный вес).

### Рассмотренные альтернативы

- **Dependency injection в стиле aiomisc (`@inject`)**: Реестр сервисов, поиск по типу и инъекция при старте. → **Отклонено** на данном этапе: в Go нет event loop, где сервисы "становятся доступными" асинхронно. Constructor injection (явная передача зависимостей в `New()`) — идиоматичный Go и уже достаточен. Документировать паттерн, а не строить фреймворк.

- **Reflection-based реестр сервисов**: Глобальный map имя → экземпляр, заполняемый при регистрации. → **Отклонено**: вводит глобальное мутабельное состояние, усложняет тестирование, противоречит философии явной конфигурации.

- **Сторонние библиотеки пула горутин (`ants`, `pond`)**: Готовые реализации с бо́льшим набором функций. → **Отклонено** для начальной реализации WorkerPool: текущий модуль имеет только две runtime-зависимости (prometheus, testify). Минимальный граф зависимостей — ключевое качество тулкит-библиотеки. Канал + `sync.WaitGroup` с ограничением покрывает основной сценарий.

## Анализ рисков и пробелов

### Неоднозначности требования

- **Объём «похожего на aiomisc»**: В aiomisc ~20 типов сервисов, утилит и фреймворкных функций. Какое подмножество в рамках данного проекта? PeriodicService, WorkerPool и gRPC-сервер — наиболее очевидные пробелы; DI и cron-планирование — спорные приоритеты.

- **Целевая аудитория**: Это внутренний тулкит (конвенции одной команды) или публичная open-source библиотека? Публичные библиотеки требуют гарантий стабильности API, semver-дисциплины и обратной совместимости. Внутренние могут итерировать быстрее. Это влияет на то, что выносить в публичный API, а что оставлять internal.

- **Observability новых сервисов**: HttpServer включает Prometheus-метрики из коробки. Должны ли PeriodicService, WorkerPool и GrpcServer также предоставлять встроенные метрики (счётчик выполнений, задержка, глубина очереди)? Явно не указано, но вероятно — да, для консистентности.

- **Соглашение по логированию**: HttpServer и HttpClient по умолчанию используют `slog.Default()` и принимают `WithLogger`. Закреплён ли этот паттерн для всех новых пакетов? По имеющемуся коду — да, но явной документации нет.

### Граничные случаи

- **Первый тик PeriodicService**: Должен ли callback выполниться сразу при старте или только после первого истекшего интервала? Для задач, критичных при запуске, поведение разное. Нужно явное значение по умолчанию и опция.

- **Дренаж WorkerPool при завершении**: Когда Entrypoint отменяет ctx сервиса, должен ли пул дождаться выполнения всех поставленных в очередь, но ещё не запущенных задач, или отбросить их? Дренаж гарантирует выполнение, но может нарушить таймаут shutdown.

- **Reflection в GrpcServer**: gRPC server reflection (для `grpcurl`) обычно включают в dev/staging, но не в prod. Стоит ли предоставить опцию `WithReflection()`? Вопрос не закрыт.

- **Порт 0 в тестах для GrpcServer**: HttpServer уже поддерживает это (`Addr()` возвращает реальный адрес после старта). То же должно работать для GrpcServer для изоляции тестов.

- **Несколько HTTP-серверов**: Бизнес-HTTP-сервер и HealthServer могут работать на разных портах — оба проходят через Entrypoint. Конфликтов нет, но паттерн «два сервера» должен быть явно задокументирован.

- **Ошибка PostStart-хука**: Если PostStart-хук завершается с ошибкой, текущий Entrypoint отменяет все сервисы и возвращает ошибку. Это корректно, но означает, что HealthServer, зарегистрированный в PostStart, не поднимется — оркестраторы, ожидающие readiness-эндпоинт, могут зависнуть.

### Технические риски

- **Граф транзитивных зависимостей gRPC**: `google.golang.org/grpc` тянет `google.golang.org/protobuf`, `golang.org/x/net`, `golang.org/x/text` и другие. Уже является косвенной зависимостью (через prometheus); перевод в прямую — низкий риск, но стоит предупредить потребителей, собирающих минимальные бинарники.

- **Масштабируемость паттерна `registerOrReuse`**: Текущий паттерн в httpserver работает, но при добавлении новых пакетов с метриками придётся его повторять. До того как он разрастётся — стоит вынести в общую утилиту, например `internal/prom/register.go`.

- **Отсутствие гарантий упорядочивания PostStart**: Entrypoint запускает все сервисы конкурентно и выполняет PostStart-хуки после запуска горутин, но `Start` сервиса может не завершить инициализацию (привязку порта, подключение к БД) до выполнения PostStart. Для HealthServer это значит: сервис может кратковременно сообщать «not ready» даже после PostStart. Интерфейс `Prober` решает это, но требует документирования.

- **Единственный `shutdownCtx` на Entrypoint**: `Entrypoint.Shutdown()` запускает необратимую отмену. Если `Run` вызвать дважды на одном Entrypoint (не предусмотренный сценарий, но возможный), второй `Run` сразу увидит отменённый shutdown-контекст. Нужно задокументировать как «one-shot» или явно защитить.

- **Изоляция тестов PeriodicService**: Тесты с реальным `time.Ticker` inherently time-dependent. Нужен либо injectable clock (усложняет API), либо достаточно короткие интервалы с разумными таймаутами. Существующий паттерн в `entrypoint_test.go` (короткие context timeout'ы) работает, но хрупок под нагрузкой CI.
